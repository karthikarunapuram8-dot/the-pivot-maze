FIELD NOTES · AGENTIC AI SECURITY

The Pivot Maze

Walking an agentic AI architecture, ring by ring

Karthik Arunapuram · 22 min read

I write this from a personal lab where I red-team agentic AI systems through different threat scenarios. The goal is not exploitation. The goal is to understand where these architectures actually break, and what it would take to keep them from breaking.

If I were targeting an agentic AI platform tomorrow with serious capability, I would not show up with an exploit kit. I would show up with patience, recon, and a model of every control they have built. Then I would walk the architecture, ring by ring, until something gave.

Six months of building agents on GCP for a Fortune 500 has taught me what those rings look like. They are not always there. When they are, they are not always wired correctly. And even when they are wired correctly, the chain composes.

Here is how that day unfolds.

01

POLICY

If allow / deny decisions go through an LLM, I do not need to attack the system. I attack the policy itself. I craft a prompt that reframes my request inside a context the model finds defensible. Allow / deny becomes a debate, and debates are winnable.

If they have a deterministic PDP (OPA, Cedar, real authorization logic in the deny path), I cannot talk my way through. Fine. I look for time-of-check, time-of-use gaps. If the policy evaluates an action before its arguments are finalized, I change the target after approval. The decision was correct for one object. Execution happens against another.

If the policy is evaluated at the point of execution with arguments frozen at decision time, my TOCTOU window closes. Fine.

→ I move to identity.

02

IDENTITY

If tool authorization runs against the agent's workload identity, I only need one successful prompt injection. One. Any input that reaches the agent becomes a path to any tool the agent can call. The agent becomes my privilege escalation primitive.

If they propagate caller identity through OBO tokens, the agent is no longer my pivot. Fine. I look for bearer semantics. Any token that lands in browser storage, logs, crash dumps, or any intermediary system becomes my credential. I replay it from my own infrastructure and inherit the caller's authority.

If tokens are bound to a client-held key (DPoP or equivalent proof-of-possession), replay fails because possession of the token is no longer sufficient. Fine.

→ I look for content the agent already trusts.

03

TRUSTED CONTENT

I do not attack the prompt. I write my instructions into a document, a ticket, an email thread, a shared note. When the agent's retrieval layer fetches that content for an unrelated workflow, my instructions execute inside the legitimate flow. The agent thinks it is helping. The content tells it whom to help.

If retrieved content is treated as untrusted by default (capability scoping, content provenance, an instruction firewall between retrieved data and the agent's own directives), my indirect injection does not escalate. Fine.

→ I shape what gets retrieved in the first place.

04

RETRIEVAL CORPUS

I do not need the model to believe me. I need the retriever to prefer me. I shape documents to dominate vector search: keyword density, authoritative phrasing, structure that mimics canonical sources. My poisoned record outranks the real one. The model summarizes me.

If retrieval has provenance ranking, source-trust scoring, and per-tenant corpus isolation, my poisoned record does not surface in someone else's query. Fine.

→ I plant a payload directly in long-term memory.

05

LONG-TERM MEMORY

I plant a payload now. The agent retrieves it on a future call. That is persistence inside an agentic system. Memory is patient.

If memory is per-tenant, quarantined on write, verified before promotion to the active partition, with ACL-gated reads, my payload never reaches a useful target. Fine.

→ I chain tools.

06

TOOL CHAINING

No single tool needs excessive privilege. I chain them. One tool discovers the target. Another retrieves sensitive data. A third performs the action. Each one looks reasonable in isolation. The exploit lives in the composition.

This is the defining failure mode of agentic systems, and most platforms do not yet have a model for it.

If tool calls within a session require scoped capability tokens per chain segment, with policy evaluation at each composition boundary, the chain breaks where the privileges would have compounded. Fine.

→ I move up the supply chain.

07

TOOL DEFINITIONS

If the agent loads tool definitions without verifying signatures, I swap one. Every agent that loads my version is mine. The definition tells the agent what arguments to send, how to interpret responses, what to chain. I do not need to attack the agent at all.

If tool manifests are signed and verified at the egress gateway, the swap does not stick. Fine. Signed manifests only shift the problem. I target the signing key, the build pipeline, or a maintainer. Once I sign malicious artifacts with trusted credentials, verification works exactly as designed.

If signing keys are rotated, hardware-bound, with multi-party release approval and SBOM attestation, my supply chain attack costs more than my target is worth. Fine.

→ I move to the layer that defines what gets audited.

08

AUDIT

If audit lives in the observability stack, I have options. Log rotation. Index manipulation. Quiet patience until retention on the entry I touched expires. The observability stack was built for engineers debugging, not for an adversary trying to disappear.

If audit is WORM, hash-chained, with retention scoped to compliance regime (SOX 7 year, HIPAA 6 year, FedRAMP per ATO), tampering breaks the chain. Fine. I act faster than the audit gets reviewed, or I go after the policy that defines what gets audited in the first place.

→ I reach for the governance layer itself.

★

GOVERNANCE

If policy changes, identity broker config, secret rotations, and tool registrations are not themselves audited, I do not need to attack the runtime. I rewrite the rules. I add a policy exception in my favor. I rotate a secret and replace it with one I control. I register a malicious tool.

The governance layer is the trust anchor for everything else, which makes it the prize.

If the governance layer has its own tamper-evident ledger and dual-approval on sensitive changes, the rewrite is visible and slow. Fine. My window is whatever lies between the policy change and someone noticing.

PART II · THE ARCHITECTURE, COMPONENT BY COMPONENT

Now look at the defender

You just walked the attacker's path. Here is what they were probing against, in component-level detail. Each block below maps to a layer in Figure 1.

The point of reading these is to internalize that no single component carries the platform on its own. Each one closes a specific ring of the maze. Together they make the chain expensive, which is the only thing that actually matters.

A Control Plane · Authorization

Where anything in the platform that is allowed to exist gets defined. The control plane is small on purpose. The smaller the surface, the easier it is to audit who changed what.

Agent Release Pipeline

What it is
A build-and-release path for agents. Every shipped agent has a signed manifest: prompt text, system instructions, tool list, model parameters, SBOM, the human who approved it.
Why it exists
Without a release pipeline, "the agent" in production has no canonical definition. Operators patch prompts in dashboards. Engineers swap tools in dev consoles. Drift is invisible because there is no baseline to drift from.
What breaks
You cannot answer "what is currently in production." You cannot reproduce an incident. You cannot revert a regression. Worst case, an attacker swaps a prompt or a tool through a console with no signed artifact behind it and no one notices for weeks.
Wiring
Output of the pipeline is a signed bundle. Signing key is hardware-bound. Approval is multi-party for production rollouts. The runtime refuses to load an agent whose signature does not verify against the registry.

Agent Registry

What it is
The catalog of every agent the platform will execute, with version pinning and signature verification.
Why it exists
The single source of truth that the runtime asks "is this agent allowed to run, at this version, with these tools?" before instantiating anything.
What breaks
The runtime trusts whatever artifact it loads. Drift becomes silent. Two engineers ship two different versions of the same agent with subtly different tool lists and the platform cannot tell.
Wiring
Read-only at runtime, write-gated at the control plane. Drift detection runs continuously by re-fetching what the runtime has loaded and comparing to the registry's expected hash.

Policy Authoring

What it is
The interface and storage for policy-as-code. Every authz rule, rate limit, allowed tool combination, and action approval requirement lives here.
Why it exists
Policy is configuration with the safety properties of code. It needs versioning, peer review, rollback, and a tamper-evident change log.
What breaks
Policies become dashboard tweaks made by individuals under deadline pressure. Compliance has nothing to point an auditor at. Any change to the deny path is invisible until something fires.
Wiring
Dual-approval on production changes. Policy diffs reviewed like code. Every authoring action emits an entry to the Admin Plane Audit. Policy bundles are signed and pulled by the PDP, never pushed.

Admin Plane Audit · Governance Change Ledger

What it is
A tamper-evident ledger of every change inside the governance layer itself. Policy edits, identity broker config, secret rotations, tool registrations.
Why it exists
The governance layer is the trust anchor for everything beneath it. An attacker who reaches it can quietly rewrite the rules. The ledger is the only thing that makes that visible.
What breaks
The ring 09 attack from above. An attacker with control-plane access adds a policy exception in their favor, registers a malicious tool, or rotates a secret to one they hold. Nothing detects this because the runtime sees the new policy and obeys it as intended.
Wiring
Append-only. Hash-chained. Mirrored to a separate trust domain than the rest of the platform. Reviewed on a cadence shorter than typical attacker dwell time. This is the only ledger that matters more than the runtime audit log.

B Security Services · Enforcement

Shared services every tenant agent calls, but never owns. Putting these in their own layer means a compromised tenant cannot reach inside and reconfigure them.

Identity Broker / STS

What it is
Issues short-lived tokens. Exchanges credentials at boundaries (caller token in, downstream-audience token out).
Why it exists
Agents need to act on behalf of users without being given the user's own credential. The broker is what makes on-behalf-of work safely.
What breaks
You end up using the agent's workload identity for everything (ring 02). One prompt injection becomes a path to every tool the agent can call.
Wiring
Tokens are short-lived (minutes, not hours), audience-bound (a token minted for tool X cannot be replayed at tool Y), and ideally proof-of-possession (DPoP) so a leaked token without the holder's key is useless. The broker writes every issuance to the audit sink.

Policy Decision Point · Deterministic

What it is
OPA, Cedar, or equivalent. A deterministic engine that takes (subject, action, resource, context) and returns allow or deny.
Why it exists
Authorization decisions must be reproducible. Two identical inputs must produce the same answer. An LLM-based policy decision is by definition non-deterministic and therefore manipulable (ring 01).
What breaks
Allow/deny becomes a debate the model can be talked into. The agent's reasoning becomes the security boundary. There is no boundary.
Wiring
Runs as a sidecar or local cluster, microsecond-class latency. Policies pulled from authoring, never pushed in band. Decisions logged with the full input context so an auditor can replay any decision.

Tenant Vault

What it is
Per-tenant key store. Each tenant's encryption keys are isolated by label and access policy.
Why it exists
So that even if the runtime is compromised, the blast radius is bounded to the tenant whose keys the compromised process can reach.
What breaks
Cross-tenant compromise. One bad actor in tenant A reads or decrypts data from tenant B because all keys live in one bucket the runtime can dereference.
Wiring
HSM-backed where available. Key access is policy-mediated, not just IAM-mediated. Failed access attempts feed the inline anomaly detector.

Ingress Filter · Modal Aware

What it is
The first thing every input touches. Detects prompt injection, jailbreak attempts, and cross-modal attacks (instructions hidden in images, audio, OCRed text).
Why it exists
The model is not a security boundary. Filtering must happen before content reaches the model. "Modal aware" matters because images and audio are now first-class injection vectors and a text-only filter is blind to them.
What breaks
Direct injection lands. Cross-modal injection lands. Every other ring downstream has to assume it is being attacked from inside the trust boundary.
Wiring
Multiple detectors in series, each cheap. Logs all rejected inputs to the audit sink for adversarial training data. False positives go to a triage queue, not silent drop.

Egress DLP · Modal Aware

What it is
The last thing every output touches. Strips data exfiltration attempts before content leaves the platform.
Why it exists
Even if everything else holds, a compromised agent can still try to exfil. Markdown image URLs, email side-channels, DNS side-channels all go through the egress path. This is the choke point.
What breaks
The lethal trifecta. Private data access plus exposure to untrusted content plus ability to egress equals an exfil-ready agent. Egress DLP is what closes the third leg.
Wiring
Content-aware: strips remote images from rendered markdown, detects encoded data in URLs and DNS lookups, applies a content classifier before any agent-authored email send.

C Agent Runtime · Per Tenant

Where actual work happens. The diagram shows two parallel swimlanes (Tenant B, Tenant C) to make the point: every tenant gets the same components, isolated. There is no shared-state path from one tenant's runtime into another's.

Policy Enforcement Point · Agent Gateway

What it is
A gateway that sits in front of every agent invocation and every tool call. Four jobs: authenticate the caller, authorize the action against the PDP, enforce rate limits and egress DLP, run inline anomaly detection.
Why it exists
Without a single chokepoint, security checks get scattered across application code where they get forgotten, bypassed, or wired wrong. The gateway makes the checks unmissable.
What breaks
Tools are called directly. Authentication and authorization become the application's responsibility. They will be implemented inconsistently.
Wiring
Two phases. Phase 1 at agent invocation: authenticate the user, mint scoped tokens. Phase 2 at every tool call: re-authorize against the PDP with the action and resolved arguments, run egress DLP on the response. The PDP call is on the hot path and it has to be fast (sub-millisecond local cluster).

Workload Identity

What it is
The identity the agent service itself runs as. Unprivileged by default. Obtains scoped capability tokens per call.
Why it exists
The agent process needs an identity for system operations (telemetry, health checks) without inheriting access to user data.
What breaks
You either give the agent process broad access (which becomes the privilege escalation primitive in ring 02) or you cannot deploy it at all.
Wiring
Workload identity has no standing access to user resources. All access happens via tokens minted at request time, audience-bound to the specific downstream tool.

Propagated Caller Identity

What it is
The user's identity, propagated through the agent to downstream tools via OBO tokens.
Why it exists
When the agent calls a tool on the user's behalf, the tool must see and enforce the user's permissions, not the agent's.
What breaks
Identity flattening. The agent becomes a single super-account that fronts every user. One injection through one user becomes a path to every other user's data the agent could touch.
Wiring
OBO token from the broker, audience-bound to the specific tool, exchanged at the gateway, never sent to the model itself. Tokens are short-lived (minutes) so a leaked one expires before it can be replayed at scale.

Long-Term Memory · Per Tenant

What it is
Persistent storage the agent reads and writes across sessions. Per-tenant partitioned. ACL-gated reads. Signed writes.
Why it exists
Useful agents need state. The diagram makes the per-tenant partition explicit because cross-tenant memory is the breach vector you cannot easily detect.
What breaks
Persistent memory injection. An attacker plants a payload that activates on a future session, possibly under a different user. Cleanup requires memory purge, not session reset.
Wiring
Writes go to a quarantine partition first. Promotion to the active partition requires either operator review (for high-trust workflows) or passing a content classifier (for high-throughput ones). Reads are ACL-gated against the caller's identity, not the agent's.

Tool Use Planner

What it is
The component that decides which tools the agent will call, in what order, with what arguments. Classifies the resulting plan by risk.
Why it exists
Tool chaining is the defining failure mode (ring 06). The planner is where chain-graph constraints get enforced before execution begins.
What breaks
Tool-chain confusion. The agent picks the wrong sequence of legitimate tools and arrives at an outcome no single-tool authorization would have allowed.
Wiring
The planner emits a plan as data, not as code. The plan is checked against allowed tool graphs (which sequences of tools are permitted for which user roles). The plan, the classification, and the deny decisions all go to the audit sink.

HCD Approval Gate · High-Risk Actions

What it is
Human-in-the-loop checkpoint for actions classified as high risk. Dual approval. Anomaly check.
Why it exists
Some actions cannot be automated safely no matter how good the controls upstream. Approving a payment, deleting a customer, sending an external email under the company's name. The gate forces a human touch.
What breaks
Automation bias and rubber-stamp fatigue become the only barrier between agent and irreversible action. The barrier is paper-thin.
Wiring
Approval is independent of the agent's framing: the reviewer sees a system-generated risk summary, not the agent's pitch. Dual approval for the highest-risk actions. Approval decisions go to the audit sink. If anomaly detection fires concurrently, the action is rejected even with approval.

ALA Token Forwarding · Per Chain Segment

What it is
Content-aware delegation tokens scoped to a single chain segment. The token to call tool A does not authorize calling tool B.
Why it exists
Privilege compounding across a chain is the mechanism behind tool-chain privilege escalation. Per-segment scoping breaks the compounding.
What breaks
Confused deputy at agent scale. A low-privilege subagent reaches a high-privilege resource because the token it inherited from the supervisor was over-scoped for the path it actually took.
Wiring
Tokens are minted by the broker at each chain boundary, with audience and capability scoped to exactly the next call. Re-evaluation happens at each composition boundary, not just at the entry.

Inline Anomaly Detector

What it is
Runs in the gateway path. Watches for patterns that look wrong: repeated tool failures, sudden rate spikes, unusual argument shapes, retry loops, fan-out explosions.
Why it exists
Some attacks only become visible in aggregate. Retry storms, parallel fan-out, and goal drift all show up as patterns rather than single events.
What breaks
The economics-class attacks land silently. Wallet drains. Token bloat. The agent does its job, the bill arrives Monday.
Wiring
Lightweight rules first (rate, depth, argument-size), heavier classifiers second (semantic similarity to known attack patterns). Detector decisions are logged and feed an offline anomaly judge for pattern review.

D Telemetry and Audit Router

Two things wearing a single uniform. They look similar. They are not the same and you must not let them collapse into one.

Telemetry Sink · Metrics, Ops

What it is
Logs, metrics, traces. Operations data. Used by engineers to debug.
Why it exists
You cannot run a platform you cannot observe.
What breaks
You cannot run a platform you cannot observe.
Wiring
Ephemeral retention (days, not years). High volume. Indexed for fast search. Optimized for query, not for forensic immutability.

Audit Sink · Immutable

What it is
WORM, hash-chained, retention-scoped to compliance regime (SOX 7-year, HIPAA 6-year, FedRAMP per ATO). Tamper-evident.
Why it exists
The observability stack is not a security boundary. It was built for engineers debugging, not for an adversary trying to disappear (ring 08).
What breaks
Audit and ops collapse into the same store. An attacker who can rotate logs or manipulate indexes can hide their activity inside the same system the security team relies on for forensics.
Wiring
Independent storage backend, separate trust domain from telemetry. Hash chain validated by an external job. Tampering breaks the chain. Compliance reporting reads from the audit sink, never from telemetry.

SIEM / Compliance Reporting

What it is
Downstream consumer that reads the audit sink for security operations and compliance reporting.
Why it exists
Auditors do not want to read raw logs. SOC analysts cannot triage from raw logs at scale.
What breaks
The audit sink becomes a write-only graveyard. Information is recorded, never reviewed, attacks live in the gap.
Wiring
Subscribe to the audit sink, not the telemetry sink. Alerts tuned to compliance regimes, not to engineering on-call.

➤ The Trust Boundary

Everything to the right of the trust boundary is outside the platform's control plane: client apps, external MCP tools, the tool marketplace, the adversarial eval suite.

External MCP Tools and Marketplace

What it is
Third-party tool servers and the marketplace they are sourced from. The platform consumes them; it does not own them.
Why it exists
Useful agents need access to tools the platform team did not write. The marketplace is the supply chain for that access.
What breaks
Without signed manifests, attestation, and verification at the egress gateway, the supply chain attack from ring 07 is trivial. Tool-description injection from inside a third-party schema also lands here.
Wiring
Manifests signed at the source. Verified at the egress gateway. Static analysis of tool definitions before exposure. mTLS to pinned endpoints.

Adversarial Eval Suite / Red Team Corpus

What it is
A continuously updated corpus of attacker prompts, payloads, and behavioral probes used to evaluate the platform on every release.
Why it exists
Static defenses age into irrelevance. Continuous adversarial evaluation is what keeps the rest of the architecture honest about what it actually catches.
What breaks
Without a continuously growing eval suite, the gap between what the platform thinks it catches and what an attacker can actually do widens with every model change, tool addition, and policy edit.
Wiring
Runs on every release. New CVEs and observed in-the-wild attacks are added to the corpus continuously. Failing tests block release.

The pattern, stated plainly.

Each layer has a job. Each component within a layer has a single failure mode it prevents. The composition of the layers is what makes the system survive a real adversary, not any single control.

A serious attacker does not need to break a single layer. They need to find the layer you forgot to wire correctly, and chain the ones around it. The architecture in this diagram is what closes that gap. Every component exists because removing it makes a specific ring of the maze cheap to walk through.

The Governance Audit Bus at the top is the load-bearing component nobody talks about enough. It is what audits the people who write the policies. Without it, every other control is downstream of an attacker who reached the control plane.

A pivot tree, not a vulnerability list

This is what a serious adversary's plan actually looks like. Not a list of vulnerabilities. A pivot tree. The architecture either forces the attacker into the next ring, or it does not.

The defining failure mode of agentic systems is composition.

  • No single tool needs to be over-privileged.
  • No single prompt needs to be malicious.
  • No single identity needs to be compromised.

The exploit lives in the chain.

Most agentic platforms shipping right now have one or two of these rings. The ones that will survive the next two years have all of them, wired in the right order, with the governance layer auditing itself.
